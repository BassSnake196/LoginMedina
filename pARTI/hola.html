Sección 01

¿Qué es Firebase?

Firebase es la plataforma Backend-as-a-Service (BaaS) de Google. En lugar de construir y mantener tu propio servidor, Firebase te da servidores, base de datos y autenticación listos para usar desde el navegador.

💡
¿Por qué usarlo en NEXUS? El sistema actual usa usuarios y contraseñas hardcodeados en el JS (MOCK_USERS). Firebase reemplaza eso con una base de datos real, sesiones persistentes y seguridad del lado del servidor — sin escribir un backend propio.
Sección 02

Los 4 componentes en este proyecto

NEXUS usa solo dos servicios de Firebase, pero es importante entender los cuatro conceptos que interactúan entre sí.

🔑

Firebase Authentication

Gestiona el quién. Verifica email/contraseña y devuelve un UID único por usuario. También maneja sesiones: si cierras el navegador y lo abres, sigues logueado.

Auth Service
🗄

Firestore Database

Base de datos NoSQL en tiempo real. Aquí guardamos el rol de cada usuario. Un documento por usuario, identificado por su UID de Auth.

Base de datos
🛡

Security Rules

Reglas que viven en el servidor de Firebase. Deciden quién puede leer o escribir qué documentos, aunque alguien intente saltarse el código del frontend.

Servidor
⚙️

Firebase Web SDK

La librería JavaScript que importamos en el HTML. Es el "puente" entre nuestro código frontend y los servidores de Firebase en Google Cloud.

Frontend
🎫

Custom Claims

Datos extra dentro del JWT token del usuario. Permiten incrustar el rol directamente en el token para máxima seguridad, sin leer Firestore cada vez.

Sección 03

Flujo completo de autenticación

Cuando un usuario pulsa "Iniciar sesión", esto es exactamente lo que ocurre por debajo:

📊 Diagrama de flujo — Login con Firebase
👤
Usuario
ingresa email + pass
signIn()
🔑
Firebase Auth
verifica credenciales
UID + token
🗄
Firestore
lee campo "role"
{ name, role }
🖥
Dashboard
renderiza según rol
1
signInWithEmailAndPassword Auth verifica que el email y contraseña existen. Si son incorrectos, lanza un error. Si son correctos, devuelve un objeto user con un uid único.
2
getDoc(doc(db, "users", uid)) Con el UID obtenido, buscamos en Firestore el documento del usuario para leer su nombre y su rol (admin / editor / viewer).
3
ROLE_CONFIG[role] Con el rol ya en el estado de la app, el dashboard muestra u oculta secciones, botones y estadísticas según los permisos configurados en ROLE_CONFIG.
⚠️
Auth y Firestore son dos servicios separados Auth solo sabe email + contraseña. Firestore sabe el nombre y el rol. Por eso necesitamos los dos: Auth para verificar identidad, Firestore para obtener datos del perfil.
Paso 01

Crear el proyecto en Firebase Console

1

Ir a console.firebase.google.com

Inicia sesión con tu cuenta de Google y haz clic en "Agregar proyecto". Dale un nombre (ej: nexus-app) y sigue los pasos del wizard.

2

Habilitar Authentication → Email/Password

En el menú lateral: Build → Authentication → Sign-in method → Email/Password → Habilitar. Este es el proveedor que usa NEXUS.

3

Crear Firestore Database

En el menú: Build → Firestore Database → Crear base de datos. Elige modo de producción (no prueba) para que las Security Rules estén activas desde el inicio.

4

Registrar la app web

Ve a Configuración del proyecto (⚙️) → Tus apps → Agregar app → Web (</>). Firebase te dará el objeto firebaseConfig con todas las claves necesarias.

Pegar la configuración del SDK

Al inicio del HTML hay un bloque comentado. Descoméntalo y reemplaza los valores placeholder por los reales que Firebase te da:

import { getAuth } from "https://www.gstatic.com/firebasejs/10.12.0/firebase-auth.js";

const firebaseConfig = {
  apiKey:            "AIzaSyBxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
  authDomain:        "nexus-app.firebaseapp.com",
  projectId:         "nexus-app",
  storageBucket:     "nexus-app.appspot.com",
  messagingSenderId: "123456789012",
  appId:             "1:123456789012:web:abcdef1234567890"
};

const app  = initializeApp(firebaseConfig);

🔍
¿Qué es cada clave? apiKey no es una contraseña — es un identificador público para que Firebase sepa a qué proyecto conectarse. La seguridad real la ponen las Security Rules, no esta clave. El projectId es el nombre único de tu proyecto en Firebase.
Paso 03

Estructura de Firestore

Firestore es una base de datos de documentos. Piensa en colecciones como carpetas y documentos como archivos JSON dentro. El campo role es el que controla TODO en NEXUS.

📁 Estructura de colecciones en Firestore
📁 users — colección
📄 uid-del-usuario-1 (ID = UID de Firebase Auth)
name string "Carlos Méndez"
email string "admin@nexus.com"
role string ⭐ "admin"
createdAt timestamp 2025-01-15T10:30:00Z
📄 uid-del-usuario-2
name string "Ana García"
role string "editor"
⚠️
El ID del documento debe ser el UID de Auth — así cuando el usuario hace login, hacemos getDoc("users/" + user.uid) y encontramos sus datos directamente, sin buscar.
📄 Crear usuario en Firestore al registrarse JavaScript 
import { createUserWithEmailAndPassword } from "firebase/auth";
import { doc, setDoc, serverTimestamp } from "firebase/firestore";

async function registerUser(email, password, name) {
  // 1. Crear en Firebase Auth (email + contraseña)
  const cred = await createUserWithEmailAndPassword(auth, email, password);

  // 2. Guardar perfil en Firestore con el rol por defecto
  await setDoc(doc(db, "users", cred.user.uid), {
    name:      name,
    email:     email,
    role:      "viewer",  // rol por defecto — admin lo cambia después
    createdAt: serverTimestamp()
  });
}
Paso 04

Los 3 comentarios "🔥 REEMPLAZAR"

El código tiene exactamente 3 puntos marcados con ese emoji. Aquí está qué hace cada uno y por qué es necesario:

🔥 Mapa de reemplazos — Mock → Firebase real
❌ Código actual (mock)
// Simula que ya hay sesión iniciada setTimeout(() => { showLogin(); // siempre muestra el login }, 1400);
✅ Con Firebase — onAuthStateChanged
onAuthStateChanged(auth, async user => { if (user) { // sesión persistente detectada const snap = await getDoc( doc(db, "users", user.uid) ); currentUser = { uid: user.uid, name: snap.data().name, role: snap.data().role }; openDashboard(); } else { showLogin(); } });
❌ Código actual (mock)
// Busca en objeto JS hardcodeado const mockUser = MOCK_USERS[email]; if (mockUser && mockUser.password === pass) { currentUser = { ...mockUser }; openDashboard(); }
✅ Con Firebase — signIn
const cred = await signInWithEmailAndPassword( auth, email, password ); const snap = await getDoc( doc(db, "users", cred.user.uid) ); currentUser = { uid: cred.user.uid, name: snap.data().name, role: snap.data().role }; openDashboard();
❌ Código actual (mock)
// Solo limpia la variable local currentUser = null; // no hay sesión real que cerrar
✅ Con Firebase — signOut
// Invalida el token en el servidor await signOut(auth); currentUser = null; // onAuthStateChanged se dispara // automáticamente y muestra el login
✅
¿Qué es onAuthStateChanged y por qué es tan importante? Es un listener que Firebase dispara automáticamente al cargar la página. Si el usuario ya inició sesión antes (el token JWT no ha expirado), lo detecta y abre el dashboard directamente — sin que el usuario tenga que volver a hacer login. Esto es lo que hace que la experiencia sea "profesional".
Seguridad 01

Tabla de permisos por rol

El campo role en Firestore determina qué puede hacer cada usuario. Esta tabla muestra la lógica implementada en ROLE_CONFIG:

Acción ADMIN EDITOR VIEWER Dónde se controla
Ver el dashboard ✓✓✓ Firebase Auth (login requerido)
Ver estadísticas ✓✓✓ ROLE_CONFIG → stats array
Ver reportes ✓✓✗ canSeeReports = false
Crear / editar contenido ✓✓✗ canEdit = false + Firestore Rules
Eliminar contenido ✓✗✗ canDelete = false + Firestore Rules
Panel Admin (usuarios) ✓✗✗ canManageUsers = false
Cambiar roles de usuarios ✓✗✗ Firestore Rules (write users)
Exportar datos completos ✓✗✗ canManageUsers → botón oculto
🚨
Los permisos del frontend no son suficientes Ocultar un botón en el UI es UX, no seguridad. Un usuario malicioso puede llamar directamente a la API de Firestore desde la consola del navegador. Por eso las Security Rules son obligatorias — son la seguridad real.
Seguridad 02

Security Rules de Firestore

Las Security Rules son el código que vive en los servidores de Firebase y verifica cada petición antes de permitirla. Se configuran en Firebase Console → Firestore → Reglas.

🛡 firestore.rules — Reglas de producción para NEXUS Rules v2 
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // ── Colección "users" ──────────────────────────────────
    match /users/{userId} {

      // Un usuario puede leer su propio documento
      // Un admin puede leer cualquier documento
      allow read: if request.auth.uid == userId
                  || request.auth.token.role == 'admin';

      // Solo admin puede escribir (cambiar roles, suspender)
      allow write: if request.auth.token.role == 'admin';
    }

    // ── Colección "content" ────────────────────────────────
    match /content/{docId} {

      // Cualquier usuario autenticado puede leer
      allow read: if request.auth != null;

      // Admin y editor pueden crear/editar
      allow write: if request.auth.token.role in ['admin', 'editor'];

      // Solo admin puede eliminar
      allow delete: if request.auth.token.role == 'admin';
    }

    // Todo lo demás: acceso denegado por defecto
  }
}
🔍

request.auth.uid

El UID del usuario que hace la petición, verificado por Firebase. No se puede falsificar — viene del token JWT firmado.

🎫

request.auth.token.role

El campo role dentro del Custom Claim del token. Más seguro que leer Firestore porque el token ya lo lleva incorporado.

🚫

Denegar por defecto

Si ninguna regla hace allow, Firestore rechaza la petición. Siempre empieza denegando y permite solo lo necesario.

Seguridad 03

Custom Claims — Seguridad máxima

En producción, el rol no debería vivir solo en un documento de Firestore que el usuario podría intentar editar. Los Custom Claims lo incrustan directamente en el token JWT firmado por Google.

🗄 Solo Firestore (básico)

⚡Fácil de implementar
⚡No requiere backend
⚠️Lee Firestore en cada login
⚠️Reglas más complejas
⚠️Cambio de rol tarda

🎫 Custom Claims (producción)

✓Rol en el token JWT
✓Reglas más simples
✓No requiere leer DB
✓Máxima seguridad
⚡Requiere Admin SDK
⚙️ Cloud Function o servidor Node.js — Asignar rol Node.js 
const admin = require("firebase-admin");

// Esta función corre en TU servidor, nunca en el frontend
async function assignRole(uid, role) {
  // Incrusta el rol en el JWT del usuario
  await admin.auth().setCustomUserClaims(uid, { role: role });

  // El cliente necesita refrescar su token para que tome efecto
  // Puedes forzarlo con: await auth.currentUser.getIdToken(true)
}

// Ejemplo de uso
await assignRole("uid-de-carlos", "admin");
await assignRole("uid-de-ana",    "editor");
await assignRole("uid-de-luis",   "viewer");
⚠️
El Admin SDK solo corre en el servidor Nunca incluyas el Admin SDK ni sus claves de servicio en el código del frontend. El código de frontend usa el SDK del cliente (firebase/auth), el servidor usa el Admin SDK (firebase-admin).
Seguridad 04

Las capas de seguridad

La seguridad bien implementada tiene varias capas. Si una falla, las demás protegen. Aquí están las capas del sistema NEXUS de menor a mayor fortaleza:

1

UI — Elementos ocultos o deshabilitados

Los botones "Eliminar", el panel Admin y el tab de Reportes solo aparecen si el rol lo permite. Es la primera línea, pero no es seguridad real — es UX.

2

Lógica JS — ROLE_CONFIG y canEdit / canDelete

El objeto ROLE_CONFIG controla qué stats, tabs y acciones renderiza el código. Protege contra uso accidental, pero un developer con las DevTools puede saltárselo.

3

Firebase Authentication — Token JWT

Sin un token válido, Firestore rechaza todas las peticiones. Nadie sin login puede leer ni escribir datos. Esta capa sí es real — vive en el servidor de Firebase.

4

Security Rules — Verificación server-side

Aunque alguien tenga un token válido (está logueado), las Rules verifican que su rol le permita hacer esa operación específica. Esta es la capa más importante.

5

Custom Claims — Rol incrustado en el JWT

Para producción: el rol está firmado criptográficamente en el token por Google. Es imposible falsificarlo. Es la capa más robusta pero requiere un backend con Admin SDK.

🎯
Resumen práctico para empezar Para un MVP funcional, implementa las capas 1–4. La capa 5 (Custom Claims) es para cuando el producto ya tiene usuarios reales y necesitas seguridad de nivel empresarial. Las capas 3 y 4 juntas son suficientes para la mayoría de los proyectos.
