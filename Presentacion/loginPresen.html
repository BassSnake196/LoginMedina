01 — Introducción

¿Qué es Firebase?

Firebase es una plataforma de desarrollo de Google que provee infraestructura backend lista para usar. En lugar de construir tu propio servidor de autenticación, base de datos y almacenamiento, Firebase te da APIs que consumes directamente desde el frontend.

🔐

Authentication

Gestiona usuarios: registro, login, logout, recuperación de contraseña. Soporta email/password, Google, GitHub, etc.

🗄️

Firestore

Base de datos NoSQL en tiempo real. Aquí guardamos los datos del usuario: nombre, rol, preferencias, etc.

🛡️

Security Rules

Reglas declarativas que controlan quién puede leer o escribir qué en Firestore, directamente desde la consola.

💡
Dato clave

Firebase Authentication y Firestore son servicios separados que trabajan juntos. Auth identifica al usuario (¿quién eres?), Firestore guarda sus datos y permisos (¿qué puedes hacer?).

03 — Authentication

Cómo funciona la autenticación

Firebase Authentication usa tokens JWT (JSON Web Token) — un estándar seguro de la industria. El token viaja en cada petición y Firebase lo verifica automáticamente.

🪪 ¿Qué es un JWT?

Es una cadena codificada en base64 dividida en 3 partes: header (algoritmo), payload (datos del usuario: uid, email, custom claims) y signature (firma criptográfica que Firebase verifica).

⏱️ Vida del token

El token de Firebase dura 1 hora. El SDK lo renueva automáticamente en segundo plano cuando está por expirar. No necesitas hacer nada — onAuthStateChanged siempre tiene el token fresco.

05 — Registro

Registro de usuario

Al registrarse, se crean dos cosas: una cuenta en Firebase Auth y un documento en Firestore. Son registros separados que se vinculan por el uid.

🔐 En Firebase Auth

Guarda solo las credenciales de acceso: email, contraseña (hasheada, nunca en texto plano), y el uid único del usuario. No guarda nombre, rol ni ningún dato extra.


uid: "abc123xyz"
email: "juan@ucol.mx"
password: █████ (hash)
emailVerified: false

🗄️ En Firestore

Guarda los datos del perfil: nombre, rol, fecha de registro. El documento se crea justo después del registro, usando el uid como ID del documento.


uid: "abc123xyz"
name: "Juan Pérez"
email: "juan@ucol.mx"
role: "user"
createdAt: Timestamp
registro.js JavaScript 
// 1. Crear cuenta en Firebase Auth
const { user } = await createUserWithEmailAndPassword(auth, email, password);
 
// 2. Crear documento en Firestore con el uid como ID
await setDoc(doc(db, 'users', user.uid), {
  uid:       user.uid,
  name:      nombre,
  email:     email,
  role:      'user',        // rol por defecto — admin lo cambia después
  createdAt: serverTimestamp()
});
 
// 3. Redirigir al dashboard
window.location.href = 'dashboard.html';
🔑
¿Por qué usar el uid como ID del documento?

Porque así getDoc(doc(db, 'users', user.uid)) siempre encuentra el documento del usuario en sesión sin necesitar una query. Es la forma más eficiente y la que usan las Security Rules para verificar permisos.

06 — Sesión

Manejo de sesión activa

onAuthStateChanged es el observer central de Firebase. Se ejecuta cada vez que el estado de autenticación cambia: al hacer login, al hacer logout, y al cargar la página (para restaurar sesiones guardadas).

auth-guard.js JavaScript 
import { onAuthStateChanged } from "firebase/auth";
import { doc, getDoc } from "firebase/firestore";
 
onAuthStateChanged(auth, async (user) => {
 
  if (!user) {
    // No hay sesión → redirigir al login
    window.location.href = 'index.html';
    return;
  }
 
  // Hay sesión → leer rol de Firestore
  const userDoc = await getDoc(doc(db, 'users', user.uid));
  const role = userDoc.data()?.role ?? 'user';
 
  // Redirigir según rol
  if (role === 'admin') {
    window.location.href = 'admin.html';
  } else {
    window.location.href = 'dashboard.html';
  }
 
});

07 — Firestore

Estructura de datos en Firestore

Firestore organiza datos en colecciones → documentos. Cada documento es un objeto JSON con campos. Para el sistema de usuarios, la estructura es:

📁 users (colección)
  📄 abc123xyz (documento — ID = uid del usuario)
    uid: "abc123xyz"
    name: "Juan Pérez"
    email: "juan@ucol.mx"
    role: "user" // o "admin"
    createdAt: Timestamp(2025-01-15)

  📄 xyz987abc
    ...
Operaciones CRUD básicas JavaScript 
// LEER un documento
const snap = await getDoc(doc(db, 'users', uid));
const data = snap.exists() ? snap.data() : null;
 
// ESCRIBIR / CREAR un documento
await setDoc(doc(db, 'users', uid), { name, role: 'user' });
 
// ACTUALIZAR un campo específico
await updateDoc(doc(db, 'users', uid), { role: 'admin' });
 
// LEER todos los documentos de una colección
const snapshot = await getDocs(collection(db, 'users'));
snapshot.forEach((doc) => console.log(doc.data()));
08 — Roles y permisos

Sistema de roles (RBAC)

Firebase no tiene un sistema de roles incorporado — tú lo construyes guardando el rol en Firestore y verificándolo en cada pantalla. Este patrón se llama RBAC (Role-Based Access Control).

👤

Rol: user

Usuario estándar. Puede ver su propio perfil. No puede acceder a rutas de administración ni leer datos de otros usuarios.


✓ Ver su perfil ✗ Panel admin ✗ Ver otros users
👑

Rol: admin

Administrador. Puede leer todos los usuarios, cambiar sus roles y acceder a cualquier ruta protegida del sistema.


✓ Ver su perfil ✓ Panel admin ✓ Cambiar roles
Acción Sin sesión user admin
Ver página de login ✓ ✗ ✗
Ver dashboard propio ✗ ✓ ✓
Ver panel admin ✗ ✗ ✓
Leer datos de otro usuario ✗ ✗ ✓
Cambiar rol de un usuario ✗ ✗ ✓
⚠️
Los roles del frontend son solo UI

Deshabilitar botones o redirigir al usuario en JS no es suficiente seguridad. Alguien puede abrir la consola y saltarse esas restricciones. La seguridad real está en las Security Rules de Firestore — si el servidor rechaza la petición, no importa lo que el usuario haga en el navegador.

09 — Custom Claims

Custom Claims vs Roles en Firestore

Hay dos formas de implementar roles en Firebase. Cada una tiene sus casos de uso:

🏷️

Custom Claims

El rol viaja dentro del JWT token. No necesitas leer Firestore — el claim está disponible instantáneamente. Requiere Admin SDK (Node.js en servidor) para asignarlos.


✓ Muy rápido ✓ Seguro ✗ Necesita backend Máx 1000 bytes
🗄️

Rol en Firestore

El rol se guarda como campo en el documento del usuario. Se lee con getDoc después del login. Más simple: no necesita backend.


✓ Sin backend ✓ Fácil de cambiar ✗ Un extra read Recomendado
Custom Claims — requiere Admin SDK (Node.js) Node.js 
// Solo se puede ejecutar desde un servidor con Admin SDK
const admin = require('firebase-admin');
 
// Asignar claim "admin" a un usuario
await admin.auth().setCustomUserClaims(uid, { role: 'admin' });
 
// En el frontend, leer el claim del token
const tokenResult = await user.getIdTokenResult();
console.log(tokenResult.claims.role); // "admin"
Para proyectos sin backend (solo frontend + Firebase), usa roles en Firestore. Para proyectos con un servidor Node.js/PHP que ya usa Admin SDK, los Custom Claims son más eficientes porque no requieren una lectura extra a Firestore en cada carga de página.
10 — Security Rules

Reglas de seguridad de Firestore

Las Security Rules son la última línea de defensa. Se configuran en Firebase Console → Firestore → Rules. Incluso si alguien manipula el frontend, el servidor rechaza cualquier petición que no cumpla las reglas.

🚨
Nunca uses estas reglas en producción

Las reglas por defecto de Firebase permiten lectura y escritura a cualquiera. Siempre configura reglas antes de lanzar tu app.

firestore.rules — reglas comentadas Rules 
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
 
    match /users/{userId} {
 
      // Cualquier usuario autenticado puede leer su PROPIO documento
      allow read: if request.auth != null
                   && request.auth.uid == userId;
 
      // Al registrarse: puede crear su propio doc, pero NO asignarse "admin"
      allow create: if request.auth != null
                     && request.auth.uid == userId
                     && request.resource.data.role == 'user';
 
      // Puede actualizar su doc, pero NO cambiarse el rol
      allow update: if request.auth != null
                     && request.auth.uid == userId
                     && request.resource.data.role == resource.data.role;
 
      // Admin puede leer CUALQUIER usuario y cambiar roles
      allow read, write: if request.auth != null
                          && get(/databases/$(database)/documents/users/$(request.auth.uid))
                            .data.role == 'admin';
    }
 
  }
}

🔍 request.auth

Contiene la información del usuario que hace la petición. request.auth.uid es el UID verificado — no se puede falsificar.

📄 resource.data

Los datos actuales del documento en Firestore. request.resource.data son los nuevos datos que se intentan escribir.

🔗 get()

Permite leer otro documento dentro de una rule. Aquí se usa para verificar que el usuario que hace la petición tenga role == 'admin'.

11 — Rutas protegidas

Proteger páginas según rol

Cada página HTML verifica el estado de autenticación al cargarse usando onAuthStateChanged. Esto previene que usuarios no autorizados vean contenido protegido.

Patrón guard para página de admin JavaScript 
onAuthStateChanged(auth, async (user) => {
 
  // Sin sesión → login
  if (!user) { window.location.href = 'index.html'; return; }
 
  // Leer documento de Firestore
  const snap = await getDoc(doc(db, 'users', user.uid));
 
  // No existe el doc o no es admin → redirigir
  if (!snap.exists() || snap.data().role !== 'admin') {
    window.location.href = 'dashboard.html';
    return;
  }
 
  // ✓ Autenticado + es admin → mostrar contenido
  document.getElementById('loading').style.display = 'none';
  document.body.classList.add('ready');
 
});
Archivo Condición para entrar Redirige a si falla
index.html No tener sesión activa dashboard.html o admin.html
dashboard.html user != null (cualquier rol) index.html
admin.html user != null y role == 'admin' dashboard.html
✅
Doble protección

El guard en JS protege la experiencia de usuario. Las Security Rules en Firestore protegen los datos en el servidor. Aunque alguien desactive el JS del navegador, no puede leer datos de otros usuarios porque Firestore rechaza la petición.

12 — Setup

Cómo crear el proyecto en Firebase

1

Ir a console.firebase.google.com

Inicia sesión con tu cuenta de Google y haz clic en "Crear un proyecto". Dale un nombre (ej: cms-ucol) y decide si quieres Google Analytics.

2

Agregar una app web

En la pantalla principal del proyecto, haz clic en el ícono </> (Web). Registra la app con un nombre. Firebase te dará el objeto firebaseConfig con tus claves.

3

Activar Authentication

En el panel izquierdo → Build → Authentication → Get started. En la pestaña Sign-in method, habilita Email/Password.

4

Crear la base de datos Firestore

Panel izquierdo → Build → Firestore Database → Create database. Elige "Start in test mode" (después cambias las rules a producción). Selecciona la región más cercana.

5

Configurar Security Rules

En Firestore → pestaña Rules. Pega las reglas de la sección anterior y haz clic en Publish.

6

Crear el primer admin manualmente

Regístrate normalmente desde tu app. Luego en Firestore Console → colección users → tu documento → edita el campo role de "user" a "admin". Ya puedes entrar al panel de admin.

Importante: type="module"

Los scripts que usen import/export de Firebase necesitan <script type="module">. Sin eso, el navegador no puede procesar los imports y obtendrás un error de sintaxis.
